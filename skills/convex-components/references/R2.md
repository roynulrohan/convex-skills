# Cloudflare R2 Storage Setup & Management

This document explains how R2 (Cloudflare's S3-compatible object storage) is configured and managed in this project, including automatic cleanup mechanisms and guidelines for adding new R2-backed features.

## Overview

We use the official [@convex-dev/r2](https://www.convex.dev/components/cloudflare-r2) component for Convex to handle file uploads, serving, and deletion. The setup includes:

- **Client-side uploads** via signed URLs
- **Automatic cleanup** when files are replaced
- **Periodic orphan cleanup** via daily cron job
- **Extensible architecture** for future R2-backed features

## Configuration

### Environment Variables

The following environment variables must be set in your Convex deployment:

```bash
npx convex env set R2_TOKEN <your-token>
npx convex env set R2_ACCESS_KEY_ID <your-access-key-id>
npx convex env set R2_SECRET_ACCESS_KEY <your-secret-access-key>
npx convex env set R2_ENDPOINT <your-endpoint>
npx convex env set R2_BUCKET <your-bucket-name>
```

### Convex Component Setup

The R2 component is registered in `convex/convex.config.ts`:

```typescript
import r2 from "@convex-dev/r2/convex.config.js";
const app = defineApp();
app.use(r2);
```

## Backend Client Setup

```typescript
// convex/r2.ts
import { R2 } from "@convex-dev/r2";
import { components } from "./_generated/api";
import { DataModel } from "./_generated/dataModel";

export const r2 = new R2(components.r2);

// Required: Export client API for frontend uploads
export const { generateUploadUrl, syncMetadata } = r2.clientApi<DataModel>({
  // Optional: Validate upload permissions
  checkUpload: async (ctx) => {
    const identity = await ctx.auth.getUserIdentity();
    if (!identity) throw new Error("Unauthorized");
  },
  // Optional: React to completed uploads
  onUpload: async (ctx, key) => {
    // e.g., associate key with a record
    console.log("File uploaded:", key);
  },
});
```

## How It Works

### 1. File Uploads

The `clientApi()` call above exports `generateUploadUrl` and `syncMetadata` — these are required by the React hook:

```typescript
import { useUploadFile } from "@convex-dev/r2/react";
import { api } from "../convex/_generated/api";

const uploadFile = useUploadFile(api.r2);
const imageKey = await uploadFile(file);
```

The returned `imageKey` is a unique identifier stored in your database (e.g., `campaigns.imageKey`).

### 2. Serving Files

Files are served via signed URLs generated by `getR2Url()`, which wraps `r2.getUrl()` with a default 1-day expiry to reduce URL churn from Convex reactive queries:

```typescript
import { getR2Url } from "./r2";

// In a query or mutation — uses 1-day expiry by default
const imageUrl = campaign.imageKey ? await getR2Url(campaign.imageKey) : null;

// Custom expiry (in seconds)
const shortLivedUrl = campaign.imageKey
  ? await getR2Url(campaign.imageKey, { expiresIn: 3600 })
  : null;
```

> **Always use `getR2Url()` instead of `r2.getUrl()` directly.** The extended expiry prevents image flickering caused by Convex's real-time reactivity generating new signed URLs on every query re-run.

### 3. Automatic Cleanup on Replacement

When a campaign's `imageKey` is updated (e.g., user uploads a new cover image), a **trigger** automatically schedules deletion of the old image:

```typescript
// In convex/campaignStats.ts
triggers.register("campaigns", async (ctx, change) => {
  if (change.operation !== "update") return;

  const oldImageKey = change.oldDoc.imageKey;
  const newImageKey = change.newDoc.imageKey;

  if (oldImageKey && oldImageKey !== newImageKey) {
    await ctx.scheduler.runAfter(0, internal.r2.deleteR2Object, {
      key: oldImageKey,
    });
  }
});
```

### 4. Periodic Orphan Cleanup

A daily cron job scans all R2 objects and deletes any that aren't referenced in the database:

```typescript
// In convex/crons.ts
crons.daily(
  "cleanup-orphaned-r2-images",
  { hourUTC: 3, minuteUTC: 0 },
  internal.r2.cleanupOrphanedImages,
  {},
);
```

This catches:

- Files from failed/abandoned uploads
- Files that somehow weren't cleaned up by triggers
- Any other orphaned files

## Adding New R2-Backed Features

When adding a new feature that stores files in R2 (e.g., profile pictures, attachments), follow these steps:

### Step 1: Add the Field to Your Schema

```typescript
// convex/schema.ts
users: defineTable({
  // ... other fields
  profileImageKey: v.optional(v.string()), // R2 object key
});
```

### Step 2: Register the Table's Keys for Cleanup

Update `getAllUsedR2Keys` in `convex/r2.ts` to include keys from your new table:

```typescript
export const getAllUsedR2Keys = internalQuery({
  args: {},
  returns: v.array(v.string()),
  handler: async (ctx) => {
    const usedKeys: Array<string> = [];

    // Existing: Campaign cover images
    const campaigns = await ctx.db.query("campaigns").collect();
    for (const campaign of campaigns) {
      if (campaign.imageKey) {
        usedKeys.push(campaign.imageKey);
      }
    }

    // NEW: Profile pictures
    const users = await ctx.db.query("users").collect();
    for (const user of users) {
      if (user.profileImageKey) {
        usedKeys.push(user.profileImageKey);
      }
    }

    return usedKeys;
  },
});
```

### Step 3: Add a Trigger for Automatic Cleanup (Optional but Recommended)

If users can replace their files, add a trigger to clean up old files immediately:

```typescript
// In convex/campaignStats.ts or a new triggers file
triggers.register("users", async (ctx, change) => {
  if (change.operation !== "update") return;

  const oldKey = change.oldDoc.profileImageKey;
  const newKey = change.newDoc.profileImageKey;

  if (oldKey && oldKey !== newKey) {
    await ctx.scheduler.runAfter(0, internal.r2.deleteR2Object, {
      key: oldKey,
    });
  }
});
```

### Step 4: Use in Your Queries/Mutations

```typescript
import { getR2Url } from "./r2";

// Get signed URL for display (1-day expiry by default)
const profileImageUrl = user.profileImageKey
  ? await getR2Url(user.profileImageKey)
  : null;
```

## Available R2 Methods

The `r2` instance (from `@convex-dev/r2`) provides these methods:

| Method                                  | Context               | Description                                                    |
| --------------------------------------- | --------------------- | -------------------------------------------------------------- |
| `r2.getUrl(key, options?)`              | Query/Mutation/Action | Get a signed URL for serving a file                            |
| `r2.deleteObject(ctx, key)`             | Mutation/Action       | Delete a file from R2                                          |
| `r2.listMetadata(ctx, limit?, cursor?)` | Query                 | List file metadata with pagination                             |
| `r2.getMetadata(ctx, key)`              | Query                 | Get metadata for a single file                                 |
| `r2.store(ctx, blob, options?)`         | Action                | Store a file from server-side code                             |
| `r2.clientApi<DataModel>({...})`        | Setup                 | Export `generateUploadUrl` + `syncMetadata` for client uploads |

### `r2.store` Options

```typescript
await r2.store(ctx, blob, {
  key: "my-custom-key", // Optional: custom object key (default: auto-generated)
  type: "image/jpeg", // Optional: MIME type override
});
```

These internal functions in `convex/r2.ts` support the cleanup system:

| Function                | Type            | Purpose                                        |
| ----------------------- | --------------- | ---------------------------------------------- |

## Best Practices

1. **Always store the R2 key**, not the full URL. URLs are signed and expire.

2. **Use `getR2Url()`** (not `r2.getUrl()` directly) to generate URLs. It applies a 1-day default expiry that reduces flickering from Convex reactive query re-runs.

3. **Add new tables to `getAllUsedR2Keys`** immediately when adding R2-backed features to prevent accidental deletion.

4. **Use triggers for immediate cleanup** when files can be replaced, rather than waiting for the daily cron.

5. **Don't hardcode key prefixes** in the cleanup logic - it should work for all keys regardless of naming convention.

## Frontend Image Stabilization

R2 signed URLs change on every Convex query re-run (even for the same object), which causes image flickering. The codebase has a two-layer fix:

### Backend: Extended URL Expiry

`getR2Url()` in `convex/r2.ts` defaults to 1-day expiry. This means previously-generated URLs remain valid even when the client holds a cached version.

### Frontend: URL Stabilization

**`LqipImage`** (for cover images, thumbnails) — has built-in path comparison. When only the URL signature changes (same path, different query params), it skips the loading state reset. No extra work needed.

**`StableAvatarImage`** (for profile pictures in avatars) — drop-in replacement for `AvatarImage` from Radix. Use in `.map()` callbacks and `ColumnDef` cells where hooks can't be called directly:

```tsx
import { StableAvatarImage } from "@repo/ui";

// Inside .map() or ColumnDef — works where useStableUrl hook can't
<Avatar>
  <StableAvatarImage src={user.profileImageUrl} alt={user.username} />
  <AvatarFallback>{initials}</AvatarFallback>
</Avatar>;
```

**`useStableUrl` hook** — for direct use in components (not inside `.map()`):

```tsx
import { useStableUrl } from "@repo/ui";

const stableUrl = useStableUrl(user.profileImageUrl);
<AvatarImage src={stableUrl} />;
```

### When to Use What

| Context                            | Solution                                              |
| ---------------------------------- | ----------------------------------------------------- |
| `LqipImage` component              | Nothing extra — handles it internally                 |
| Avatar in a component body         | `useStableUrl` hook + `AvatarImage`                   |
| Avatar inside `.map()` / ColumnDef | `StableAvatarImage` (drop-in)                         |
| Plain `<img>` inside `.map()`      | Create a small wrapper component using `useStableUrl` |

## Troubleshooting

### Files being deleted unexpectedly

Check that the table's R2 keys are registered in `getAllUsedR2Keys`. If a key isn't in that list, the cleanup job will delete it.

### Orphaned files accumulating

The cleanup cron runs daily at 3 AM UTC. If you're seeing orphans, check:

1. Is the cron job running? Check the Convex dashboard.
2. Are triggers registered for tables where files can be replaced?

### Upload failures

Verify:

1. Environment variables are set correctly
2. CORS policy on the R2 bucket allows your domain
3. User is authenticated (check `checkUpload` callback)
