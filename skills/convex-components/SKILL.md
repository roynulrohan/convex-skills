# Using Convex Components

Convex components are installable packages that add **prebuilt, sandboxed backend features** to an app — each ships its own tables, functions, and isolated execution, and wires into your app through a small public API.

**Core principle: don't reinvent the wheel.** Before building a backend capability from scratch — or before reaching for a raw third-party SDK — check whether a component already solves it. There are 100+ and the set grows constantly. The live directory below is the source of truth for *what exists* and *how each one works*. Prefer it over memory: component lists and APIs go stale fast, and guessing a component's surface is a common source of bugs.

## The Component Directory — start here

Two machine-readable indexes, regenerated by Convex and always current:

- **Index (scan this first):** https://www.convex.dev/components/llms.txt
  A flat list of every published component. Each entry: name, `Category`, a one-line `Description`, the npm package, and a `Markdown` link to that component's own doc. Compact and greppable — ideal for "what's available."
- **Category view:** https://www.convex.dev/components/components.md
  The same catalog grouped by category (auth, payments, storage, durable functions, messaging, database, AI, integrations, collaboration, backend, and more), with install commands inline. Easier to browse by area.

Every entry links to a **per-component `.md`** (e.g. `…/rate-limiter/rate-limiter.md`) — that component's full, current spec: install, version, weekly downloads, API, and usage examples. **Fetch that file whenever you actually work with a component.** It is more reliable and more up to date than anything you remember.

## Two workflows

### 1. Discovery — is there a component for this problem?

When planning a feature or weighing how to solve a problem, *before* writing it from scratch:

1. Fetch the index (`llms.txt`) and scan it — browse by `Category`, or grep for keywords that match the problem (`batch`, `stripe`, `cron`, `email`, `rate`, `geospatial`, `rag`, …).
2. If a candidate fits, open its per-component `.md` to confirm scope and fit.
3. Recommend the component over a hand-rolled solution or a raw SDK, and explain the tradeoff.

The most common reason an agent reinvents the wheel is simply not knowing a component exists. A cheap scan of the index prevents it. Examples of the judgment to make: *"User wants to process a large dataset in chunks → there's a batch/workpool component for exactly this."* *"User is wiring the Stripe SDK by hand → there's a Stripe component that already handles webhooks and subscription sync."*

### 2. Depth — work with a component the project already uses

When a task touches a component the app already depends on (you'll see it in `convex/convex.config.ts` as `app.use(...)`, or as a `@convex-dev/*` / community entry in `package.json`):

1. Identify the package name.
2. Get its per-component `.md` — search `llms.txt` for the package to find the exact `Markdown` URL — and fetch it.
3. Work from that spec: its real API, options, and patterns. **Don't guess.** Each component's surface is specific, and the per-component doc is the contract.

## How components work — the shared model

The directory documents each component on its own but assumes you know the model every component shares. This is the part the catalog doesn't teach:

### Install and wire

```typescript
// convex/convex.config.ts
import { defineApp } from "convex/server";
import componentName from "@convex-dev/<component-name>/convex.config";

const app = defineApp();
app.use(componentName);
export default app;
```

Run `npx convex dev` to generate code, then access the component through the generated `components` object:

```typescript
import { components } from "./_generated/api";
const instance = new ComponentClass(components.componentName, { /* config */ });
```

### Named instances

The same component can be installed multiple times under different names — useful when you need several independent configurations:

```typescript
app.use(componentName, { name: "scrapeQueue" }); // → components.scrapeQueue
```

### Transaction semantics (the key gotcha)

Component mutations **participate in the calling mutation's transaction**:

```typescript
export const doWork = mutation({
  handler: async (ctx) => {
    await ctx.db.insert("myTable", { data: "value" });
    await component.doSomething(ctx); // same transaction
    // if this mutation throws, BOTH writes roll back
  },
});
```

Catch the component call to roll back **only** its writes while letting the parent continue:

```typescript
try {
  await rateLimiter.limit(ctx, "myLimit", { throws: true });
} catch (e) {
  // only the component's writes roll back; the parent mutation proceeds
}
```

### Keep derived state in sync with triggers

Components that maintain derived data (e.g. aggregates, counters) stay correct when you drive them from table changes using `convex-helpers` triggers (see the `convex-helpers` skill):

```typescript
import { Triggers } from "convex-helpers/server/triggers";
import { customCtx, customMutation } from "convex-helpers/server/customFunctions";

const triggers = new Triggers<DataModel>();
triggers.register("myTable", aggregate.trigger());
const mutation = customMutation(rawMutation, customCtx(triggers.wrapDB));
```

### Testing and dashboard

Many components ship a test helper and expose their tables in the Convex dashboard under a component dropdown (each component's tables are isolated). Testing setup varies per component — check the component's `.md` / README for its exact `convex-test` wiring rather than assuming a universal pattern.

## Vetting a component before adopting it

The catalog mixes **official** and **community** packages side by side — judge before you adopt:

- **Official `@convex-dev/*`** (author `get-convex`): maintained by the Convex team. Safe default.
- **Community components**: vet with the signals the directory exposes — **weekly npm downloads** and **version** (both shown in the per-component `.md`), plus the linked GitHub repo. A one-person experiment and a battle-tested package look identical in the list until you check those numbers; they're your fastest quality signal.

## Best practices

- **Name instances for their job** (`emailWorkpool`, `scrapeQueue`), not `workpool1` — you'll often run several instances of one component.
- **Configure once** when constructing the instance, not per call.
- **Use triggers** to keep component-derived state in sync automatically rather than updating it by hand in every mutation.
- **Respect runtime limits** (e.g. free-tier concurrent-function caps) when a component fans out work.
